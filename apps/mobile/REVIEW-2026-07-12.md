## 1. The blocker is now a CODE bug, not config (N1)

**Your mobile authorize request never includes a zkLogin `nonce`.** We ran the
full flow live (simulator → Google → bridge → exchange): Google's code swap
returns **200**, `verifyGoogleJwt` passes, and then `deriveAddress` fails with
`EnokiClientError 400 { code: "salt_failure" }`. Decoded claims of the mobile
id_token: `iss/aud/azp/sub/email/at_hash` — **no `nonce`**.

Compare the web flow you mirror (`packages/auth/src/client.ts` `startLogin`,
~L146–165): it creates an **ephemeral Ed25519 keypair**, calls Enoki
`createZkLoginNonce` (binds key + maxEpoch), and puts that nonce in the Google
auth URL. Enoki's salt service only derives for JWTs carrying an Enoki-issued
zkLogin nonce. Your `src/auth/google.ts` (~L39–48) builds the auth URL with
PKCE but no nonce → every derive 400s.

**Fix (funkii's `SPEC_AUDRIC_MOBILE.md` Fork C documents exactly this):**
1. Before opening the browser: generate ephemeral keypair (RN-safe path),
   `POST /v1/zklogin/nonce` to Enoki (needs `EXPO_PUBLIC_ENOKI_API_KEY` in
   mobile env — public key, safe to inline).
2. Add `nonce=<enoki nonce>` to the authorize URL in `google.ts`.
3. Persist the ephemeral secret + randomness + maxEpoch (SecureStore) — B2's
   zkLogin signing needs the same material, so this is not throwaway.
4. Mysten reference: `MystenLabs/react-native-zklogin-poc` (2024 POC, Expo).

Verify step: sign in on the simulator → web-v3 dev log shows the derived
address instead of `salt_failure`.
