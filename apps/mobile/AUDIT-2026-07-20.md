## 1. CRITICAL — the chat fabricates a wallet balance

**Files:** `src/app-state/store.tsx:218`, `:226-277` · `src/components/chat/conversation.tsx:110-123`, `:183-213` · `src/app-state/catalog.ts:109-113`

Any chat message matching `/balance|usdc|wallet|hold|spend|money/`
(`store.tsx:218`) is intercepted client-side by `classify()`. It never reaches
the model or `/api/balance`. `send()` routes non-`"text"` kinds to a fake
1600 ms `setTimeout` (simulated "thinking"), then injects `buildDemoReply("wallet")`
— a hardcoded string at `store.tsx:230-240`:

> "You're holding 124.50 USDC that's spendable, plus 0.82 SUI reserved for gas.
> Want me to send some?"

Rendered alongside it: `WalletCard` (`conversation.tsx:183-213`), which hardcodes
`124.50` USDC and `0.82` SUI **inline in JSX** — not even sourced from
`catalog.ts` — using the same badge/typography as the genuinely-live wallet tab.

`metadata.demo` (`conversation.tsx:110-123`) only selects *which* fake card to
render. **It never labels anything as fake.** There is no visual distinction
between this and a real assistant turn.

### Why this is worse than ordinary mock data

1. **The app steers users into it.** The first empty-state chip a new user sees
   is `"Check my balance"` → `"What's my USDC balance?"` (`catalog.ts:110`).
   The literal first tap produces a fabricated balance.
2. **It is money-adjacent and Send is real.** A user reads "124.50 USDC
   spendable", believes it, and opens a Send sheet that broadcasts genuine
   signed transactions.
3. **It compounds with #2.** After session expiry the app still presents as
   signed-in, so an effectively-logged-out user is told they hold 124.50 USDC.
4. **It contradicts `CLAUDE.md` rule 6** (honesty / no overclaim) — the repo's
   own governing constraint.

The *code comments* are honest ("prototype's canned assistant reply", "these
surfaces are still mock"), so this was a known scaffold. Nothing user-facing
carries that caveat.

`WorklogCard` (`conversation.tsx:148-181`) compounds it further: hardcoded
`"Worked for 8s"` / `"· 3 steps"` with fake checkmarks, rendered in the same
slot where genuine turns show the real `CotTimeline`. Same surface alternates
between fully real and fully fabricated step logs.

Server-side confirmation that no wallet tool exists to back any of this:
`src/app/api/chat+api.ts:150-152` wires only `web_search`, with an explicit
comment that web-v3's wallet tools "need a session + Neon/blob/Enoki and, for
wallet, the Phase-0 gate — added later".

## 2. HIGH — expired sessions restore as signed-in

**Files:** `src/auth/useAuth.tsx:58-68`, `:106` · `src/lib/identity.ts:33` · `src/lib/wallet/keys.ts:39`

*(Found by Codex.)*

The launch effect loads the stored session and sets status `"signed-in"` with no
expiry check:

```tsx
const [s, pref] = await Promise.all([loadSession(), loadLockPref()]);
setSession(s);
setStatus(s ? "signed-in" : "signed-out");
```

`expiresAt` is written at `useAuth.tsx:106` and read in exactly one place —
`identity.ts:33` `expiresLabel()`, which only *displays* it in Settings. It is
never enforced. `isProofExpired` (`keys.ts:39`) guards the wallet **keys**, not
the session token.

**Behavior after the server's 7-day cap:** app opens signed-in → every data
route 401s → balance resolves null. Send fails closed (`sui == null` ⇒
`insufficient`), which is correct. But the user sits in a signed-in shell with
no working data and no prompt to re-authenticate — and per #1, the chat still
reports a confident fake balance.

## 3. HIGH — Send dead-ends on mainnet

**File:** `src/lib/wallet/send.ts:47`

```ts
if (SUI_NETWORK !== "testnet") {
  throw new Error("Real Send is testnet-only pending Phase-0 parity.");
}
```

Nothing in the UI reflects this gate. On mainnet the Send button is fully live:
the user enters a recipient, enters an amount, taps Send — and only then eats
the error. The gate belongs at the surface, not at the bottom of the money path.

Underlying parity work (separate, larger): web-v3 sends **gasless USDC** via
`balance::send_funds`; mobile sends **native SUI**. Mainnet parity means
implementing the former.

## 6. LOW — `limit` clamps to 1

**File:** `src/app/api/transactions+api.ts:44-47`

```ts
const rawLimit = Number(url.searchParams.get("limit"));
const limit = Number.isFinite(rawLimit) ? Math.min(Math.max(1, ...), MAX_LIMIT) : DEFAULT_LIMIT;
```

`Number(null)` is `0`, and `Number.isFinite(0)` is `true` — so a **missing**
`limit` takes the clamp branch and resolves to `1`, never `DEFAULT_LIMIT`.
Latent only because `lib/wallet-data.ts:91` always passes one explicitly.

## 11. MEDIUM — dev builds showed a fake second "Continue with Google"

**File:** `src/app-state/store.tsx` (`ONBOARD_START_STEP`)

`const ONBOARD_START_STEP = __DEV__ ? 0 : 1;` — step 0 is the prototype's *mock*
Google welcome. It existed so the full 4-step flow stayed previewable behind the

effect was: dev users sign in for real, reach the app, and are then shown a fake
sign-in button. Leftover from the bypass removal. Onboarding now always starts at
step 1 (Privacy).

## 12. ~~BLOCKER~~ RESOLVED 2026-07-21 (environment, not code) — the AI Gateway key was revoked

**Resolved:** a fresh key was supplied and written to both `.env.local` files
(gitignored, never committed). Verified 200 on `/v1/chat/completions`, and smoke check
#1 now passes on-device — see "Check #1 closed" in `FIX-PLAN-2026-07-20.md`.
The original finding, for the record:

`AI_GATEWAY_API_KEY` in `apps/mobile/.env.local` (byte-identical to web-v3's) is
rejected by the gateway: **HTTP 401** on both `/v1/models` and
`/v1/chat/completions`. Every chat turn fails with
`GatewayAuthenticationError`, including the pre-existing `web_search`.

Pre-existing and unrelated to this audit's changes, but it blocks end-to-end
verification of the new `balance_check` tool (see the smoke-test results). **A fresh
key must be minted** (Vercel → AI Gateway → API Keys) and set in BOTH
`apps/mobile/.env.local` and `apps/web-v3/.env.local`.
